# 架构总览

## 项目目标

本项目面向报废机动车回收企业的自营业务，核心目标是打通这条链路：

1. 客户在线了解服务并提交报废预约
2. 内勤或客服确认资料、报价、安排拖车
3. 外勤或拖车人员在移动端推进现场执行
4. 客户持续查看订单进度并保持联系

## 为什么当前先收口为一个移动端加后端

本项目天然存在三类使用场景：

- `apps/mobile` 的 `customer` 分区：客户自助提交与查询
- `apps/mobile` 的 `operator` 分区：移动执行、拍照、备注、快捷推进状态
- 未来的 `apps/admin-web`：PC 后台处理、筛选、审核、配置

当前仓库优先把客户侧和移动执行侧收口在 `apps/mobile`，是因为首期最需要跑通“客户提交 -> 后端处理 -> 现场执行 -> 客户看进度”的移动闭环。这样可以在不额外维护多个前端工程的前提下，把业务对象、路由分区和接口协议先打稳；等 PC 重操作后台真正进入实现，再单独补 `apps/admin-web` 会更顺手。

## 后端定位

`apps/backend` 是业务真相源，负责统一定义：

- 客户、车辆、订单、估价、附件、时间线等核心对象
- 订单状态和值域
- 状态流转规则
- 后台账号、角色、权限
- 上传资料和操作留痕

三端都基于后端提供的同一套业务口径工作，不各自维护一份“自己的真相”。

## 共享层定位

推荐共享包包括：

- `packages/shared-types`
  - DTO、状态枚举、字段字典、公共类型
- `packages/api-sdk`
  - 请求封装、接口定义、按角色区分的 API 调用入口
- `packages/shared-utils`
  - 与平台无关的纯函数和基础工具

当前真正已落地的共享层是 `packages/shared-types`。其余共享包只有在两端以上稳定复用时再补，避免在启动期过早抽象。

## 当前推荐技术路线

- `apps/mobile` 使用 `Vue 3 + Vite + TypeScript`
- 后端使用 `Koa + TypeScript + MySQL`
- 客户侧当前先以 H5 方式开发联调
- 等客户主链路稳定后，再评估是否将 `apps/mobile` 的客户侧迁移到 `uni-app`
- `apps/admin-web` 作为后续扩展，不占用当前移动闭环优先级
