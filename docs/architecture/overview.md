# 架构总览

## 项目目标

本项目面向报废机动车回收企业的自营业务，核心目标是打通这条链路：

1. 客户在线了解服务并提交报废预约
2. 内勤或客服确认资料、报价、安排拖车
3. 外勤或拖车人员在移动端推进现场执行
4. 客户持续查看订单进度并保持联系

## 为什么采用三端一后端

本项目天然存在三类使用场景：

- `customer-h5`：客户自助提交与查询
- `admin-web`：PC 后台处理、筛选、审核、配置
- `admin-h5`：移动执行、拍照、备注、快捷推进状态

如果把这些场景硬塞进一个前端项目，角色、路由、页面复杂度会很快缠在一起。拆成三端后，页面和交互边界更清楚，但又因为它们共享同一套业务对象和状态语义，所以最适合放在一个 `monorepo` 里统一管理。

## 后端定位

`apps/backend` 是业务真相源，负责统一定义：

- 客户、车辆、订单、估价、附件、时间线等核心对象
- 订单状态和值域
- 状态流转规则
- 后台账号、角色、权限
- 上传资料和操作留痕

三端都基于后端提供的同一套业务口径工作，不各自维护一份“自己的真相”。

## 共享层定位

推荐共享包包括：

- `packages/shared-types`
  - DTO、状态枚举、字段字典、公共类型
- `packages/api-sdk`
  - 请求封装、接口定义、按角色区分的 API 调用入口
- `packages/shared-utils`
  - 与平台无关的纯函数和基础工具

共享层的目标是统一协议和稳定能力，不是把所有逻辑都抽走。应用内仍然可以保留面向各自页面流程的 service、store 和视图逻辑。

## 当前推荐技术路线

- 前端三端首期统一使用 `Vue 3 + Vite + TypeScript`
- 后端使用 `Koa + TypeScript + MySQL`
- 客户端先以 H5 方式开发联调
- 等客户主链路稳定后，再评估是否将 `customer-h5` 迁移到 `uni-app`
