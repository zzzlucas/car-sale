# 客户预约闭环设计

## 目标

把当前移动端“看起来能跑”的估价预约流程，改成真正由后端支撑的最小闭环：

- 客户端补齐预约表单缺失字段
- 车辆照片走真实后端上传
- 预约提交写入后端真实存储
- 预约记录、进度详情改成读取后端真实数据
- 地图先采用轻量方案：地址文本 + 经纬度 + 浏览器定位入口，暂不接第三方地图 SDK

## 设计原则

- 先保证 `v1` 闭环可体验，不把范围扩到小程序正式地图接入、支付、复杂状态编排
- 保持现有 `customer` 端路由结构，避免无关页面大改
- 后端继续沿用当前 `Midway + TypeORM` 模式，不把业务逻辑塞进 controller
- 在无登录态的前提下，用浏览器本地 `visitorKey` 作为“我的订单”过滤依据，避免匿名用户互相看到订单

## 表单范围

本轮统一落地以下字段：

- `vehicleType` 车辆类型
- `brandModel` 品牌型号
- `plateNumber` 车牌号
- `plateRetention` 是否保留原车牌
- `wheelMaterial` 轮毂材质
- `weightTons` 整备质量（吨）
- `contactName` 联系姓名
- `contactPhone` 联系电话
- `pickupAddress` 取车地址
- `pickupLatitude` 取车纬度
- `pickupLongitude` 取车经度
- `vehiclePhotos` 车辆照片 URL 列表

## 后端方案

新增预约订单实体，直接保存客户提交时的预约快照，不拆分成多张业务表。原因：

- 当前阶段重点是先把体验闭环跑通
- 订单状态流转还未扩展到运营端，不需要过早拆复杂模型
- 预约页、记录页、进度页需要的字段都可以由单表 + timeline JSON 支撑

订单服务负责：

- 生成订单号
- 保存匿名访客标识 `visitorKey`
- 写入初始状态 `submitted`
- 构造 timeline
- 按 `visitorKey` 返回“我的订单”
- 详情/进度查询时校验 `visitorKey`

## 前端方案

预约页改为真实表单页：

- 增加联系人、电话、地址、重量、轮毂材质、车牌号、照片上传
- 增加“读取当前位置”按钮，成功后回填经纬度
- 图片通过后端上传接口换取 URL，再随订单一起提交
- 提交失败不再静默回退 mock，而是向用户展示错误

记录页与进度页：

- 只展示后端返回的数据
- 去掉硬编码的时间、用途、订单号占位
- 详情页补展示联系人、地址、重量、轮毂材质、照片等预约信息

## 风险与边界

- 本轮不接入第三方地图 SDK，因此没有正式地图选点面板；先保留地址 + 经纬度能力，后续可替换成腾讯地图/小程序选点
- 当前匿名模式依赖浏览器本地 `visitorKey`，换浏览器或清理本地存储后，“我的订单”不会自动同步
- 后端依赖现有 TypeORM/MySQL 运行环境；若本地数据库不可用，本轮至少保证单测与类型检查通过
