# 高德 Key 池地址搜索设计

## 目标

在当前 `apps/mobile` 客户预约页里补上可用的地址搜索能力，并把高德接入收口到 `apps/backend`，支持多个 Key 顺序兜底，避免把 Key 暴露到前端。

## 当前约束

- 项目仍处于低成本验证阶段，优先保证“能搜索地址并回填经纬度”。
- 现有客户预约表单已经有 `pickupAddress`、`pickupLatitude`、`pickupLongitude` 字段。
- 当前不做完整嵌入地图，不引入地图页，不追加复杂前端 SDK 初始化逻辑。
- Key 可能来源不稳定，因此要把高德调用集中在单一后端服务里，便于后续替换、补 Key、关停某个 Key。

## 方案结论

采用“后端高德代理 + Key 池兜底 + 前端轻量地址搜索”的方案。

### 后端

- 在 `apps/backend/src/modules/app/service` 新增地图服务。
- 通过 `AMAP_WEB_SERVICE_KEYS` 读取多个 Key，支持逗号、空白符或换行分隔。
- 首期只封装“地址建议搜索”能力，调用高德 Web 服务 `place/text`。
- 每次请求从当前游标 Key 开始尝试；命中限额、鉴权失败或网络错误时切换到下一个 Key。
- 对前端仅返回本项目需要的最小字段：
  - `id`
  - `name`
  - `formattedAddress`
  - `latitude`
  - `longitude`

### 前端

- 在客户预约页保留现有详细地址输入框。
- 新增“搜索地址建议”按钮，用户主动触发检索，减少无意义调用量。
- 搜索成功后展示建议列表；点击任一结果，回填地址和经纬度。
- 保留现有“读取当前位置”逻辑，不在本轮追加逆地理编码。

## 配置约定

- `AMAP_WEB_SERVICE_KEYS`：Key 池主入口。
- `AMAP_WEB_SERVICE_KEY`：单 Key 兼容入口；当未配置池时兜底读取。
- `AMAP_WEB_SERVICE_TIMEOUT_MS`：服务端请求超时，默认走轻量值。

## 错误处理

- 未配置任何 Key：后端返回“地图服务暂未配置”。
- 高德全量 Key 都失败：返回“地图服务暂时不可用，请稍后重试”。
- 关键词太短：前后端都不发起请求，直接返回空结果或提示。
- 没有命中地址：前端提示用户换成小区、写字楼、道路名再搜。

## 测试策略

- 后端服务测试：
  - Key 池解析
  - 首个 Key 失败后自动切换下一个 Key
  - 关键词过短时不触发高德请求
- 前端服务测试：
  - 关键词过短时不请求后端
  - 正常请求 `/app/map/address-suggestions`
- 页面源契约测试：
  - 预约页包含“搜索地址建议”“推荐地址”“自动回填经纬度”等关键 UI 入口
