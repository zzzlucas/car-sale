# 报废车预约平台文档体系设计

## 背景

本项目是一个面向报废机动车回收企业自营业务的 `monorepo` 仓库，目标是同时承载三类前端和一个统一后端：

- `customer-h5`：客户预约报废车、查看进度、联系客服的 H5 端
- `admin-web`：内勤/客服/管理人员使用的 PC 运营后台
- `admin-h5`：外勤/拖车/现场人员使用的移动运营端
- `backend`：统一的 `Koa + MySQL` 业务后端

当前仓库还处于启动期，首要任务不是把治理体系铺满，而是先沉淀一套足够清晰、能直接指导建仓和后续开发协作的 `AGENTS.md` 与 `docs` 骨架。

## 目标

本轮输出的文档体系需要满足以下目标：

1. 明确仓库的技术路线、目录结构和协作边界。
2. 固定 `v1` 的业务范围，避免项目初期范围膨胀。
3. 明确三端职责差异，防止页面、接口和状态语义混乱。
4. 给“先 H5、后评估 `uni-app` / 小程序”的路线留出架构空间。
5. 保持文档轻量，优先服务真实开发，而不是制造维护负担。

## 非目标

本轮不做以下事情：

- 不直接初始化业务代码或脚手架。
- 不建立完整的测试、部署、CI、需求治理自动化体系。
- 不假设当前仓库已经具备 `pnpm dev`、`pnpm build`、`pnpm test` 等可执行入口。
- 不把项目设计成多租户 SaaS 平台。
- 不把客户端前端从一开始就绑定到 `uni-app`。

## 项目定位与 `v1` 范围

### 业务定位

项目定位为“业主/回收企业让客户预约报废车服务的平台”，核心是打通客户提交预约、运营处理订单、外勤推进现场流程、客户查看进度这条闭环。

### `v1` 业务目标

- 客户能够通过 H5 端了解服务、提交车辆信息、发起预约、查看进度、联系客服。
- 内勤/客服能够通过 PC 后台查看预约、确认资料、更新估价、安排拖车、推进状态。
- 外勤/拖车人员能够通过移动端查看待办任务、联系客户、上传现场资料、更新执行状态。

### `v1` 暂不纳入

- 在线支付
- 政务或车管所系统自动直连
- 多企业多租户 SaaS 能力
- 复杂 CRM、营销裂变、分销体系
- 一开始就正式建设微信小程序运行时

## 架构决策

### 1. 采用“三端一后端”的 `monorepo`

采用 `monorepo` 而不是多个独立仓库，原因如下：

- 三个前端共享同一套订单、状态、附件、用户与进度模型。
- 后续会存在共享类型、API SDK、状态枚举和字段字典，集中管理更稳。
- 项目初期开发节奏快，单仓更适合统一约束和快速联调。

### 2. 后端是唯一业务真相源

`apps/backend` 负责定义和维护以下核心真相：

- 订单主数据
- 统一状态枚举与流转规则
- 时间线留痕
- 附件与材料元数据
- 后台账号、角色、权限

三端只消费后端定义的业务语义，不各自发明状态或字段口径。

### 3. 前端先用 `Vue 3 + Vite + TypeScript`

当前推荐路线：

- `customer-h5`：先使用 `Vue 3 + Vite + TypeScript` 开发并编译为 H5
- `admin-web`：使用 `Vue 3 + Vite + TypeScript`
- `admin-h5`：使用 `Vue 3 + Vite + TypeScript`

这样可以优先获得更轻的本地开发、联调与调试体验，先把业务链路和接口协议打稳。

### 4. 后续按需评估 `uni-app`

未来如果确认需要微信小程序，优先评估把 `customer-h5` 演进为或迁移到 `uni-app`。为此，本轮文档会明确以下边界：

- 业务规则不要写死在页面组件内部。
- 接口类型、状态枚举、字段字典统一放在共享层。
- 上传、登录态、订单状态、时间线结构由后端统一定义。
- 客户端避免过早依赖强浏览器专属交互。

### 5. `admin-h5` 是移动运营补充端，不是 `admin-web` 的完整镜像

`admin-web` 负责重操作与全量管理：

- 列表筛选
- 详情审核
- 估价确认
- 状态流转
- 配置维护

`admin-h5` 负责现场和轻操作：

- 今日任务
- 一键联系客户
- 快捷更新状态
- 现场拍照与备注
- 附件上传

这样可以避免把 PC 后台复杂度原样复制到移动端。

## 三端职责设计

### `customer-h5`

聚焦客户自助链路：

- 首页与品牌信任展示
- 车辆估价/预约表单
- 我的预约
- 进度查询
- 流程说明、常见问题、在线客服

### `admin-web`

聚焦运营后台链路：

- 预约单列表与筛选
- 客户资料、车辆资料、附件资料查看
- 估价确认与预约安排
- 状态流转与时间线记录
- 基础配置与后台管理

### `admin-h5`

聚焦移动执行链路：

- 待办任务查看
- 电话联系与导航
- 现场拍照
- 执行备注
- 状态快捷推进

## 核心数据对象

文档会以以下对象为基础讲述产品和接口：

- `customer`
  - 客户基础身份与联系方式
- `vehicle`
  - 车辆品牌、型号、车牌、VIN、整备质量、手续信息、照片资料
- `scrap_order`
  - 报废预约主单，关联客户、车辆、来源、预约信息、当前状态
- `valuation`
  - 估价记录，区分系统初估和人工确认估价
- `pickup_task`
  - 外勤/拖车任务，含时间、地点、负责人、执行备注
- `order_timeline`
  - 订单关键动作与状态时间线，是进度展示的真相源
- `attachment`
  - 车辆照片、证件照、现场凭证、回收证明、注销证明等
- `operator_user`
  - 后台和移动运营端账号，至少区分管理员、内勤/客服、外勤

## 统一状态语义

文档会固定一套统一状态，三端共享，不允许各端自行命名不同但语义相近的状态：

- `submitted`
- `contacted`
- `quoted`
- `scheduled_pickup`
- `picked_up`
- `dismantling`
- `deregistration_processing`
- `completed`
- `cancelled`

页面展示文案可以按端做友好化处理，但底层状态语义必须统一。

## 各文档职责

### `AGENTS.md`

根入口，只放执行层规则与默认协作边界：

- 默认中文沟通
- `monorepo` 与技术栈说明
- 三端一后端的基本事实
- 先 H5 后评估 `uni-app` 的前端策略
- 哪些改动默认直接执行，哪些必须先确认
- 最小验证口径

### `docs/README.md`

作为文档导航页，告诉后续开发者先看哪里，不承担大段规则说明。

### `docs/architecture/*`

说明结构、目录、边界与技术决策：

- 为什么是三端一后端
- 为什么是 `monorepo`
- 为什么先 `Vue 3 + Vite H5`
- 如何给未来小程序接入留边界

### `docs/product/*`

说明客户链路和运营链路：

- 客户视角的预约与进度体验
- 后台与移动运营端的分工边界

### `docs/requirements/*`

保持轻量需求记录：

- `index.md` 作为索引
- `project.md` 记录当前项目活跃需求和 `v1` 决策

### `docs/development/*`

沉淀工程协作约定：

- 什么时候在端内闭环，什么时候抽共享
- API 约定
- 类型共享约定
- 目录说明

### `docs/testing/*`

只写最小验证基线，不伪装成已经成熟的测试体系。

### `docs/deployment/*`

先提供轻量占位说明，只描述未来会如何沉淀环境与部署文档，不假装现阶段已经存在部署流程。

## 文档风格要求

本轮文档遵循以下风格：

- 默认中文
- 轻量、直接、可执行
- 优先回答“做什么、为什么、入口在哪、边界在哪”
- 避免过多空泛原则
- 避免抄大型成熟项目的重治理结构

## 验证要求

由于当前仓库尚未初始化完整工程，本轮验证以文档级验证为主：

1. 检查所有新增文档路径与相互引用是否合理。
2. 检查文档中的目录、应用名、共享包名保持一致。
3. 明确区分“当前已存在”与“后续目标结构”，避免误导。
4. 不声称尚未建立的脚手架、命令或环境已经可用。

## 风险与约束

### 风险 1：文档先行但脚手架未建

文档中需要明确“目标结构”而不是“现状结构”，避免后续读者把规划误读成已实现事实。

### 风险 2：三端边界失焦

如果 `admin-h5` 设计成 `admin-web` 的完整镜像，后续容易出现重复页面和维护负担。因此文档必须明确移动端是补充执行端。

### 风险 3：过早跨端

如果过早按 `uni-app` 主基座约束所有前端，会拖慢启动期迭代。因此文档需要固定“先 H5、后评估迁移”的原则。

## 结论

本轮设计的核心结论如下：

1. 本项目采用“三端一后端”的 `monorepo` 结构。
2. 前端首期统一采用 `Vue 3 + Vite + TypeScript`，优先编译和联调 H5。
3. 客户端后续如需小程序，优先在业务模型稳定后评估 `uni-app`。
4. 文档体系采用中等治理强度，重点沉淀协作边界、架构说明、产品链路、需求记录和开发约定。
5. `admin-web` 与 `admin-h5` 明确分工，避免移动端复制 PC 后台。
